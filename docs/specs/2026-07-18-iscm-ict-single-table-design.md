# ISCM_ICT 로그 3테이블 → 1테이블 통합 설계

- 작성일: 2026-07-18
- 대상 설비: ISCM_ICT (eTR5001SII ICT 검사설비)
- 상태: 승인됨, 구현 대기

## 배경 / 목적

현재 ISCM_ICT 로그 1파일은 3개 섹션으로 나뉘어 3개 테이블에 분산 저장된다.

| 섹션 | 테이블 | 행수 |
|------|--------|------|
| `[KEY]:VALUE` 헤더블록 72종 | `LOG_ISCM_ICT` | 1 |
| `// Header_Data` CSV 23필드 | `LOG_ISCM_ICT_HDATA` | 1 |
| `// Component_Data` CSV 12필드 | `LOG_ISCM_ICT_COMP` | N (파일당 ~1221행) |

3테이블 분리 + BARCODE 조인 구조가 조회/운영에 번거로워, **단일 테이블 `LOG_ISCM_ICT`** 로 통합한다.

- **Header_Data(`_HDATA`) 섹션은 완전히 폐기**한다.
- **헤더블록(`[KEY]:VALUE`)에서 5종 그룹만 추출**하여 Component 각 행 앞에 붙인다.
- Component_Data가 본문(N행)이 되고, 헤더 값은 매 행 반복된다.

## 추출 대상 (헤더블록 → 컬럼)

실제 로그(`TEST_ISCM_6711001761B_20260711124534_OK.csv`) 기준 확정.

| 컬럼 | 헤더 키 | 예시 값 |
|------|---------|---------|
| `BARCODE` | `BARCODE : [ 6711001761B ]` | `6711001761B` |
| `TOTAL_RESULT` | `[RESULT]` | `PASS` |
| `RESULT_CURRENT` | `[RESULT CURRENT   ]` | `PASS - 133.237 uA` |
| `HEX_FILE_1` ~ `HEX_FILE_16` | `[HEX FILE  1]` ~ `[HEX FILE 16]` | `TC364_...FU00.hex` (후행 `/` 제거) |
| `CRC_1` ~ `CRC_16` | `[HEX CRC32  1]` ~ `[HEX CRC32 16]` | `4DBAFBC2` |

- 헤더블록의 나머지 키(MODEL/DEVICE/DOWNLOAD/HEX BOM 등)는 **버린다**.
- HEX FILE / HEX CRC32는 로그에 각각 16개 슬롯이 있으며 대부분 빈칸이다. **16컬럼씩 그대로** 저장한다(빈 슬롯은 빈 문자열).
- 키 정규화: VRL이 `upcase(replace(key, r'\s+', "_"))` 하므로 `HEX FILE  1` → `HEX_FILE_1`, `HEX CRC32  1` → `HEX_CRC32_1`. `HEX_CRC32_N` 값을 컬럼 `CRC_N` 에 담는다.

## Component_Data (본문 N행)

12필드 CSV. 컬럼명은 기존 `LOG_ISCM_ICT_COMP` 와 동일하게 유지한다.

`SEQ_NO`(1부터 증가), `PART_NAME`, `ACT_V`, `STD_V`, `MSR_V`, `HLIM`, `LLIM`, `HPIN`, `LPIN`, `PART_TYPE`, `RESULT`, `DEV`, `SKIP_FLAG`

- `RESULT`(부품별 판정, 10번째 필드) 와 헤더의 `TOTAL_RESULT` 를 이름으로 구분한다.

## 최종 테이블 스키마 (`LOG_ISCM_ICT`)

DDL 빌더 규약(`oracle-ddl.ts`)의 컬럼 순서를 따른다.

```
-- SYSTEM_COLUMNS
LOG_ID           NUMBER GENERATED BY DEFAULT AS IDENTITY  NOT NULL  -- PK
EQUIPMENT_ID     VARCHAR2(50)  NOT NULL
LINE_CODE        VARCHAR2(50)
LOG_TIMESTAMP    TIMESTAMP DEFAULT SYSTIMESTAMP
FILE_NAME        VARCHAR2(500)

-- 헤더 추출 컬럼 (매 행 반복)
BARCODE          VARCHAR2(100)
TOTAL_RESULT     VARCHAR2(50)
RESULT_CURRENT   VARCHAR2(100)
HEX_FILE_1  .. HEX_FILE_16    VARCHAR2(500)   -- 16개
CRC_1       .. CRC_16         VARCHAR2(100)   -- 16개

-- Component 컬럼 (행별)
SEQ_NO           VARCHAR2(100)
PART_NAME        VARCHAR2(500)
ACT_V            VARCHAR2(500)
STD_V            VARCHAR2(500)
MSR_V            VARCHAR2(500)
HLIM             VARCHAR2(500)
LLIM             VARCHAR2(500)
HPIN             VARCHAR2(500)
LPIN             VARCHAR2(500)
PART_TYPE        VARCHAR2(500)
RESULT           VARCHAR2(50)
DEV              VARCHAR2(500)
SKIP_FLAG        VARCHAR2(500)

-- SYSTEM_TAIL_COLUMNS
IS_LAST          VARCHAR2(1) DEFAULT 'N'
IS_SAMPLE        VARCHAR2(1) DEFAULT 'N'
CREATED_AT       TIMESTAMP DEFAULT SYSTIMESTAMP

-- 트리거 전용 (자동생성기가 만들지 않음 → DDL 에 명시)
ACTUAL_DATE / SHIFT_CODE / ZONE_CODE   -- LOG_ICT 컬럼 정의 복사

CONSTRAINT PK_LOG_ISCM_ICT PRIMARY KEY (LOG_ID)
```

## 적재 방식

기존 다중테이블 규약(`.data.TABLES`)을 버리고, 다른 설비와 동일한 **단일 테이블 경로**로 전환한다.

- VRL: `.target_table = "LOG_ISCM_ICT"`, `.data.ROWS = comp` (단일 배열).
- Backend: `LOG_ISCM_ICT` 는 이미 `BARCODE_REPLACE_TABLES` 에 등록됨 → BARCODE 단위 DELETE 후 bulk INSERT (PRESSFIT 과 동일 패턴). 누적/재전송돼도 중복 없음.
- `.data.TABLES` 다중테이블 핸들러(`format_for_api`, L1605)는 ISCM 이 유일 사용처였으나, **삭제하지 않고 휴면 상태로 남긴다** (공유 코드 수정 리스크 회피).

## 트리거

`TRG_LOG_ISCM_ICT_IS_LAST` 재생성 (기존 로직 유지):
- `ACTUAL_DATE`/`SHIFT_CODE`/`ZONE_CODE` ← `f_get_work_*(SYSDATE)` (헤더에 신뢰 가능한 날짜 없음 → SYSDATE, LOG_ICT 규약)
- `IS_LAST := 'Y'` 고정 (자기 테이블 UPDATE 없음 → bulk INSERT 가능)

**성능 주의**: 파일당 ~1221행 × `f_get_work_*` 3회 → 트리거 함수 호출 급증. 우선 유지하고, 실측 후 병목이면 트리거 내 SYSDATE 분 캐싱 등으로 최적화한다. (`LOG_ICT` 트리거가 60초 병목이었던 전례 있음.)

- 재전송 중복 없음 확인: 동일 파일 2회 투입 → BARCODE DELETE 후 재삽입으로 행수 동일.

## 결정 로그

- Header_Data 폐기, 헤더블록 5종만 추출 (사용자 지정).
- HEX FILE / CRC 16컬럼씩 그대로 (사용자 지정).
- 헤더 RESULT → `TOTAL_RESULT` 로 이름 변경하여 Component `RESULT` 와 충돌 회피 (Component 쿼리 호환 유지).
- 기존 3테이블은 DROP (백업 없이 제거, 사용자 지정 — 신규 설비라 누적 데이터 적음).
- 테이블명은 `LOG_ISCM_ICT` 유지 (registry/문서 명칭 변경 없음).
